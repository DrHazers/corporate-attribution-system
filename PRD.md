# PRD - Corporate Attribution System（当前阶段版）

## 1. 文档定位
本文档不是面向商业化产品的泛化 PRD，而是面向当前毕设阶段的项目需求说明。

目标是把“当前已经实现什么、当前默认怎么运行、下一阶段重点做什么”写清楚，方便后续继续开发、联调和答辩。

## 2. 项目目标

当前阶段的项目目标可以概括为四步：

1. 输入目标公司  
   当前稳定入口为 `company_id`，未来可以在查询层扩展到公司名称。

2. 展示控制链路图  
   以 `shareholder_structures` 为原始关系来源，基于预计算分析结果做控制链展示。

3. 输出实际控制人与国别归属  
   通过 unified control inference 生成 `control_relationships` 与 `country_attributions`。

4. 对接产业分析模块  
   在控制分析结果基础上，继续形成企业综合分析结果。

一句话总结：

> 当前项目的核心目标，是完成“企业控制分析结果 + 控制链图展示 + 产业分析接入”的研究型闭环，而不是继续扩张基础 CRUD 或无限深化控制算法。

## 3. 当前核心用户

### 3.1 研究人员
主要关注：

- 输入目标公司并查看控制链结果
- 判断企业实际控制人与国别归属
- 结合控制链结果开展产业研究

### 3.2 数据整理 / 标注人员
主要关注：

- 维护企业基础信息
- 维护主体与原始关系数据
- 对分析结果进行核对与必要的人工修订

### 3.3 项目维护者 / 开发者
主要关注：

- 保证分析结果一致
- 保证图展示与分析口径一致
- 为下一阶段产业分析模块和总展示层提供稳定接口

## 4. 当前核心功能模块

### 4.1 企业信息管理
当前已实现：

- 企业基础信息录入与查询
- 企业主表 `companies` 的基础 CRUD

当前定位：

- 为控制分析提供目标公司对象
- 为国别归属提供基础国家字段

### 4.2 主体与关系建模
当前已实现：

- `shareholder_entities` 主体建模
- `shareholder_structures` 原始关系建模
- 主体到公司 `company_id` 的映射

当前定位：

- `shareholder_entities` 是控制网络节点层
- `shareholder_structures` 是核心原始关系来源

### 4.3 控制链分析
当前已实现：

- 多层控制路径搜索
- 多路径聚合
- 股权与语义控制边混合分析
- unified control inference 作为默认主链路

当前定位：

- 是后端分析核心，不再只是“计划中能力”

### 4.4 实际控制人识别
当前已实现：

- 根据 unified 引擎候选结果识别实际控制人
- joint control 场景下不强行输出唯一实际控制人

当前定位：

- 已进入默认运行时主链路

### 4.5 国别归属推断
当前已实现：

- 基于控制链结果推断实际控制国别
- 输出 `country_attributions`
- 支持 fallback 到 `incorporation_country`

当前定位：

- 已形成与控制链结果联动的稳定输出

### 4.6 控制链图展示
当前已实现：

- 关系图数据读取
- 基于预计算结果的控制链高亮
- HTML 图构建支撑

当前定位：

- 当前主要用于演示、验证与答辩展示
- 图层默认依赖预计算结果，不承担实时重算职责

### 4.7 人工修订与结果留痕
当前已实现的范围：

- 结果表允许保留人工结果
- 普通 refresh 只覆盖自动结果
- 批量重算会保留 manual / uncertain 结果并跳过对应自动重写

当前未实现的范围：

- 完整的独立人工审核工作台
- 完整的结果版本化审计系统

### 4.8 产业分析模块
当前状态：

- 作为下一阶段重点准备接入
- 当前尚未成为主分析链路的一部分

当前定位：

- 应复用现有控制分析结果，而不是重新发明一套输入输出逻辑

## 5. 当前系统输入输出闭环

这是当前项目最重要、也最适合答辩展示的主链路：

1. 输入 `company_id`
2. 在 `shareholder_entities` 中定位目标实体
3. 基于 `shareholder_structures` 构建控制分析上下文
4. 由 unified control inference 计算控制候选、实际控制人和国别归属
5. 将结果写入：
   - `control_relationships`
   - `country_attributions`
6. 由读接口和图展示模块读取这些结果
7. 为产业分析模块提供基础输入

这里需要明确：

- `shareholder_structures` 是底层事实来源
- `control_relationships` / `country_attributions` 是分析结果快照
- 图展示默认读预计算结果，而不是临时现场重算

## 6. 当前默认行为

### 6.1 默认分析主链路
当前默认使用：

- `backend.analysis.control_inference.build_control_context()`
- `backend.analysis.control_inference.infer_controllers()`

也就是 unified control inference。

### 6.2 当前主要分析入口
当前主要入口仍然是 `company_id`。

公司名检索可以作为上层功能扩展，但不应替代当前主分析入口。

### 6.3 refresh 与读取
- refresh 入口会触发真实重算并写回结果
- GET 读取接口默认只读结果表
- 图展示依赖预计算结果

## 7. 非功能需求

### 7.1 结果一致
- 控制分析、结果读取、图展示必须尽量基于同一套主链路
- 不允许图层与分析层长期口径分叉

### 7.2 可解释
- 输出结果应带有 `basis`
- `control_type` / `attribution_type` 应足以支撑答辩解释
- 需要能说明结果来自哪类关系和路径

### 7.3 可维护
- 当前代码结构要便于继续联调
- 不因追求理想架构而破坏现有可运行主链路

### 7.4 可扩展
- 后续可扩展到 PostgreSQL
- 后续可接入产业分析模块
- 后续可增加更完整的展示层

### 7.5 适合展示
- 默认行为稳定
- 主要接口口径一致
- 图与分析结果相互支撑

## 8. 当前阶段不做的事项

当前阶段明确不优先做：

- 学术级最复杂控制模型
- 再次大规模重构核心控制算法
- 重前端工程化
- PostgreSQL 强制迁移
- 大模型接入主控制判别核心
- 无边界继续扩展特殊结构识别规则

这些内容如果后续需要推进，也应放在当前展示闭环完成之后。

## 9. 当前阶段成功标准

如果本阶段工作完成得好，系统应满足：

1. 能稳定输入公司并完成控制分析
2. 能输出实际控制人与国别归属
3. 能构建可展示的控制链图
4. 分析层、读取层、图展示层结果口径基本一致
5. 能为下一阶段产业分析模块整合提供稳定基础

## 10. 下一阶段重点

当前最推荐的开发顺序是：

1. 公司查询 / 分析入口整理
2. 控制链图展示优化
3. 产业分析模块数据接入
4. 总展示层或总接口整合
5. 答辩材料支撑性输出优化

当前阶段最重要的方向不是继续深挖控制算法，而是把已有分析能力稳定地接到展示和综合分析闭环里。
