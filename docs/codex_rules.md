# Codex 开发规则（当前阶段）

## 1. 文档目的
本文档用于约束 Codex 在当前项目中的开发方向，确保后续工作与仓库真实状态保持一致。

当前项目已经不处于“最初 CRUD 原型阶段”，也不再适合继续无边界扩张控制算法。后续开发应围绕：

- unified control inference 默认主链路
- 控制分析、读接口、图展示结果一致
- 股权链路图展示
- 产业分析模块整合
- 最终答辩展示闭环

展开。

## 2. 当前项目必须遵守的事实

### 2.1 默认分析主链路
当前默认主链路是 unified control inference：

- `backend.analysis.control_inference.build_control_context()`
- `backend.analysis.control_inference.infer_controllers()`

单公司 refresh 的默认入口是：

- `backend.analysis.ownership_penetration.refresh_company_control_analysis()`

但其默认内部实现是 unified，不是 legacy。

### 2.2 当前主要分析入口
当前稳定分析入口是 `company_id`。

如果做公司名查询，也应先完成：

1. 名称查询
2. 映射到 `company_id`
3. 再进入当前分析主链路

不要默认新建“以 company_name 为核心”的第二套分析逻辑。

### 2.3 原始事实来源与结果表
必须严格区分：

- `shareholder_structures`：核心原始关系事实表
- `control_relationships`：控制分析结果表
- `country_attributions`：国别归属结果表

禁止把 `control_relationships` 当作底层事实来源去反推控制网络。

### 2.4 当前开发数据库
当前项目开发数据库仍然是 SQLite。

可以为未来 PostgreSQL 兼容保留接口和结构意识，但不要默认推进正式迁移。

### 2.5 当前下一阶段重点
当前下一阶段重点是：

1. 输入公司并展示股权链路图
2. 对接产业分析模块
3. 完成最终展示闭环

不是继续深挖和扩大控制算法能力。

## 3. 当前开发主线

Codex 后续开发的优先级应固定为：

### 第一优先级：结果一致
- 保持 unified control inference 为默认主链路
- 保证控制分析、读接口、图展示行为一致
- 优先修复结果分叉、默认参数不一致、图和分析不一致的问题

### 第二优先级：展示闭环
- 整理公司查询与分析入口
- 优化股权链路图展示
- 保证展示层能稳定读取当前分析结果

### 第三优先级：综合分析整合
- 对接产业分析模块
- 形成控制分析 + 产业分析的综合输出

## 4. 当前算法边界

### 4.1 当前算法已经够用
当前 unified 控制分析已经能支撑：

- 多层控制链分析
- 实际控制人识别
- 国别归属推断
- 图展示支撑

因此后续默认不应把主要精力继续投入到复杂算法扩张上。

### 4.2 只允许小幅、收口式优化
允许做的优化类型：

- 默认参数统一
- refresh / recompute / 读接口 / 图层口径统一
- 明显不利于展示或答辩的小幅阈值修正
- 输出字段稳定化与文档澄清

不允许默认展开的方向：

- 随意继续扩展协议控制评分体系
- 新增复杂控制网络模型
- 新增学术化控制增强模块
- 大规模修改底层数据库结构

### 4.3 legacy 逻辑的处理原则
legacy 逻辑可以保留，但只能作为显式兼容模式存在。

不允许：

- 把 legacy 重新变成默认路径
- 在新代码里偷偷回退到 legacy
- 新增功能默认依赖 legacy 结果

## 5. Codex 后续改动原则

### 5.1 先保证结果一致，再增加功能
只要涉及以下任一内容：

- 控制分析
- 读接口
- 批量重算
- 图层展示

就必须先检查是否与 unified 默认主链路一致。

### 5.2 优先做可展示、可答辩、可联调的功能
优先事项：

- 展示链路打通
- 输出字段稳定
- 接口与图一致
- 产业分析模块可接入

低优先事项：

- 纯理论增强
- 大规模重构
- 与当前答辩闭环关系弱的底层抽象

### 5.3 不允许绕开当前默认分析主链路
新代码如果需要得到控制分析结果，应优先复用：

- `refresh_company_control_analysis()`
- `infer_controllers()`
- `analyze_control_chain_with_options()`
- `analyze_country_attribution_with_options()`

不允许新建一条默认绕开的“平行分析链路”。

### 5.4 图层不能使用与分析层不同的边过滤口径
图层与分析层必须对齐以下条件：

- `is_current`
- `is_direct`
- `effective_date`
- `expiry_date`

如果新增图层、图数据接口或图构建逻辑，必须显式检查这一点。

### 5.5 读接口默认仍然是读结果，不是实时重算
除非显式 `refresh=true` 或调用 refresh 入口，否则：

- GET 接口默认只读结果表
- 图展示默认只读预计算结果

不要默认把读接口改成“每次请求都现场重跑算法”。

### 5.6 处理手工结果时要保守
必须尊重当前结果写回规则：

- 普通 refresh 只覆盖自动结果
- 批量 recompute 要保留 manual / uncertain 结果

不要在没有明确需求的情况下改成“无条件覆盖全部结果”。

## 6. 推荐开发顺序

后续建议按以下顺序推进：

1. 公司查询 / 分析入口整理
2. 股权链路图展示优化
3. 产业分析模块数据接入
4. 总接口或总展示层整合
5. 答辩材料支撑性输出优化

只有当前面这些闭环已经比较稳定时，才考虑进一步算法细化。

## 7. 当前不建议做的事情

当前阶段不建议 Codex 主动推进：

- 大规模数据库迁移
- 再次重构核心控制算法
- 引入过重的新框架
- 复杂前端工程化
- 继续无限扩展特殊结构识别
- 把大模型接入主控制判别核心
- 为了理想分层强行重写当前目录结构

## 8. 分层与目录边界

当前仓库真实目录应按以下方式理解：

### `backend/api/`
职责：

- 暴露 HTTP 接口
- 做参数接收、错误转换、依赖注入
- 调用 `crud / analysis / visualization` 中已有能力

约束：

- 不在路由层堆放复杂控制算法
- 不在路由层直接写大段图遍历与评分逻辑

### `backend/crud/`
职责：

- 基础数据库读取与写入
- 通用实体查询

约束：

- 不在 `crud` 中实现控制推断逻辑
- 不把 `crud` 变成结果判定层

### `backend/analysis/`
职责：

- 控制链分析
- 实际控制人判定
- 国别归属推断
- refresh 与结果读取逻辑

约束：

- 是当前算法主战场
- 涉及默认行为时必须与 unified 主链路对齐

### `backend/tasks/`
职责：

- 批量重算
- 离线维护任务

约束：

- 默认行为必须与主链路一致
- 不要让 tasks 长期跑出与单公司 refresh 不一致的结果

### `backend/visualization/`
职责：

- 构图上下文整理
- 图高亮与 HTML 输出

约束：

- 负责展示，不负责重新发明一套控制判定规则
- 图层应复用已写回结果，并与分析边过滤口径一致

### `backend/models/` 与 `backend/schemas/`
职责：

- 模型与 schema 定义

约束：

- 不要因为一次小需求就大规模调整字段结构
- 结果类型字段如要调整，需同时检查兼容性、读取层和测试

### 关于 service 层
当前仓库没有强制统一的 `services/` 主干。

规则是：

- 如果只是简单编排，保持当前 `api -> crud / analysis` 结构即可
- 只有在出现明显重复编排逻辑时，才考虑新增小范围 service

不要为了理想架构强行要求当前代码迁移到完整 service 层。

## 9. 文档与测试要求

### 9.1 涉及默认行为改动时
如果修改了以下内容：

- 默认引擎
- 默认阈值
- refresh / recompute 行为
- 图层过滤规则
- 结果字段口径

则必须同步更新相应文档，优先包括：

- `README.md`
- `docs/current_control_inference_summary.md`
- `docs/ultimate_controller_algorithm_explained.md`
- 必要时补充或修订 `docs/ultimate_controller_rules_v2.md`

### 9.2 测试要求
默认只补关键测试，不泛滥加测试。

优先覆盖：

- refresh 与 recompute 是否都走 unified 默认主链路
- 图层与分析层是否用相同过滤规则
- refresh 后控制链、国别归属、图上下文是否一致
- 如果改了阈值或默认输出，是否有对应回归测试

## 10. 当前阶段的最终判断标准

Codex 的改动是否合格，优先看以下问题：

1. 是否保持了 unified 为默认主链路
2. 是否减少了分析层、接口层、图层之间的不一致
3. 是否让系统更适合“输入公司 -> 展示股权链路图 -> 接产业分析模块”
4. 是否避免了无边界继续扩张算法

如果一个改动虽然“更先进”，但会让默认链路更分裂、展示更不稳定、或者偏离当前答辩目标，那就不应优先推进。
