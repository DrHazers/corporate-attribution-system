# 产业分析分类映射规则总结

本文总结的是当前仓库里已经实现的那套“业务线文本 -> 产业分类结果”的规则主干。它对应的不是一个抽象方案，而是项目里已经能运行、能刷新到数据库里的分类流程：把 `business_segments` 表中的业务线信息，映射到 `business_segment_classifications` 表中的行业分类结果。

这份说明刻意不用太多内部代码术语，而是尽量用“本科毕业设计答辩也能讲清楚”的方式，把系统现在到底怎么判断、为什么有些能直接分类、为什么有些要保守处理讲明白。

## 1. 这套规则在做什么

这套规则的输入，是公司的一条条业务线记录，也就是 `business_segments`。一条业务线通常会有名字、别名、业务说明、收入占比、利润占比、报告期等信息。

它的输出，是 `business_segment_classifications` 里的分类结果。简单理解，就是系统会尽量回答两个问题：

1. 这条业务线大致属于哪个产业方向？
2. 这个判断到底有多稳，能不能直接当作正式结果展示？

因此，这套规则不是单纯“贴标签”，而是在做一个偏研究型的映射过程：

- 能稳定判断的，直接给出较完整的 GICS 四级分类；
- 证据还不够的，只保留到较粗层级；
- 名称太模糊、冲突太大、或者当前规则根本匹配不上时，就不强行乱分，而是进入待补判、冲突或未映射状态。

这也是它和很多“能猜就猜”的文本分类器不一样的地方。当前实现更强调“保守输出”，而不是“看起来像就硬分”。

## 2. 系统主要看哪些信息

当前规则并不是只看一个业务线名称，而是会综合几类文本来源。

### 2.1 业务线名称 `segment_name`

这是最核心的信息来源，也是权重最高的一类。很多业务线如果名字里已经带有很明确的行业特征词，系统往往可以比较快地命中规则。

比如：

- `China commerce retail business`
- `Digital media and entertainment`
- `Payment processing services`
- `Semiconductor manufacturing`

这类名字里已经包含了零售、媒体、支付、半导体等明显线索。

### 2.2 业务线别名 `segment_alias`

别名的作用通常是补充一个更“研究口径”的短名称。有些公司披露名很长、很口语化，但别名会更像真正的业务标签。

例如：

- 披露名写得很宽泛，但别名是 `Cloud infrastructure`
- 或者披露名是中文翻译，别名里有 `SaaS`、`ERP`、`CRM`

这种时候，别名往往能帮助系统把业务线从“模糊描述”拉回到更稳定的技术或行业方向。

### 2.3 业务说明 `description`

业务说明的权重低于名称和别名，但在很多场景里非常重要。因为现实中的业务线名称经常会很抽象，比如：

- `Platform services`
- `Enterprise solutions`
- `Digital ecosystem`

单看名字很难判断，但如果说明里出现了“merchant services”“cloud software”“payment gateway”“diagnostic imaging”这类词，系统就能获得更具体的证据。

### 2.4 公司上下文

当前实现还会参考公司层面的名称和简介。它不是主要证据，但能提供背景。

例如，一条业务线只写了 `Enterprise platform`，如果公司本身简介中频繁出现 software、technology、enterprise，那么系统更容易往软件/IT 服务方向理解；如果公司简介里更偏 energy、utility、power，则会影响它对能源方向的判断。

### 2.5 同公司其它业务线的辅助信息

系统还会看同一公司、同一报告期下的其它业务线文本，作为“同公司语境”的补充参考。

这一步的作用不是替代主判断，而是帮助系统在边界情形下更稳一点。比如：

- 公司其它业务线都明显属于零售、电商、物流生态；
- 当前这条业务线名字又写得比较泛；

那系统会多一个“这家公司整体可能在讲同一类商业模式”的侧面信息。

## 3. 规则是怎么一步步判断的

如果把整套流程讲成一条主线，大致可以分成下面几步。

### 3.1 先做文本标准化

系统不会直接拿原始文本硬比，而是先做一轮标准化处理，目的是把不同写法尽量拉到同一层。

例如：

- `e-commerce` 会统一成 `ecommerce`
- `cloud computing` 会统一成 `cloud`
- `artificial intelligence` 会统一成 `ai`
- `software as a service` 会统一成 `saas`
- `data centre` 会统一成 `data center`

同时，系统也会把大小写、括号、连字符、特殊符号等做统一处理。这样可以减少“其实是同一个意思，但写法不同”的干扰。

### 3.2 先判断属于哪个“大方向”

当前实现不是一上来就直接选四级行业，而是先判断业务线大概属于哪个“规则家族”。

可以把这个步骤理解成：先问“它更像哪一大类业务”，再问“这一大类里面更具体是哪一种”。

当前后端已经实现的主要家族包括：

- 数字商业 / 电商平台
- 媒体与广告技术
- 金融科技
- 软件与 IT 服务
- 半导体
- 科技硬件
- 汽车与出行
- 能源转型
- 医疗科技

这一步的好处是，系统不会因为某个局部词就直接跳到很细的结论，而是先过一层“行业大门”。

### 3.3 再在家族内部细分

如果大方向判断得比较稳定，系统会继续在这个方向里看更细的规则。

比如同样属于“软件与 IT 服务”这个大方向，还会继续区分：

- 更像应用软件
- 更像系统软件
- 更像数据处理 / 外包服务
- 更像 IT 咨询与集成

这样设计的原因是：现实里的企业业务很多都不是一句“软件”就能说明白的，系统要尽量往更具体的行业层级走，但又不能一步走得太猛。

### 3.4 评分、比较，再决定给多深

当前实现里，每条规则都会根据命中的词、命中的来源、是否出现反向词等来累积分数。

简单说：

- 明显关键词加分更多；
- 辅助关键词也会加分，但力度更小；
- 如果出现在业务线名称或别名里，权重大于出现在描述里；
- 公司上下文和同公司其它业务线也会加分，但权重更轻；
- 如果碰到明显“反向信号”，会扣分。

最后系统不是只问“有没有命中”，而是问：

- 分数够不够高？
- 是不是明显比第二候选更强？
- 有没有冲突？
- 有没有必要停在更粗层级？

### 3.5 证据不够时，宁可保守

这是当前实现里非常重要的一条原则。

如果业务线只能说明“大概属于这个方向”，但还不足以稳定落到四级行业，系统通常不会硬给最细分类，而是：

- 只保留到二级或三级；
- 进入 `pending`；
- 或者进入 `needs_llm_review`。

这就是为什么当前系统里会出现“待补充层级”“待模型补判”这类状态。它不是系统不会分，而是系统选择“不乱分”。

## 4. 什么叫“命中规则”

系统里的很多规则名，看起来像内部英文键名，例如：

- `broadline_retail_marketplace`
- `application_software`
- `data_processing_and_outsourced_services`

这些键名本质上就是“某一类业务模式的识别模板”。可以把它们理解成一组“如果文本里出现某些典型表达，就往某个产业方向判断”的规则包。

下面用几个例子说明。

### 4.1 `broadline_retail_marketplace`

这个名字可以翻译成“广义零售 / 平台交易类规则”。

它主要会看这类词：

- ecommerce
- online retail
- marketplace
- merchant platform
- shopping
- fulfillment

所以像下面这类业务线，比较容易命中：

- `China commerce retail business`
- `Online retail marketplace`

因为这些词共同指向的是“以交易撮合、线上零售、商家平台为核心”的业务模式。

### 4.2 `application_software`

这个可以翻译成“应用软件规则”。

它主要针对：

- saas
- enterprise software
- ERP
- CRM
- workflow software
- developer tools

例如：

- `Enterprise SaaS`
- `Workflow software platform`

这类业务线如果文本里给出的证据比较明确，通常会被分到软件类下更细的应用软件方向。

### 4.3 `data_processing_and_outsourced_services`

这个更适合翻译成“数据处理 / 托管 / 外包服务规则”。

它并不等于所有“云”都归这里，而是更偏向：

- data center
- colocation
- hosting services
- managed infrastructure
- outsourced services

所以像：

- `Cloud and AI infrastructure`

如果说明里更偏 data center、hosting、managed infrastructure，系统可能会往这个方向靠；但如果只有“cloud”这类宽泛说法，系统往往不会直接确认到最细层级。

### 4.4 `interactive_media_and_advertising`

这个可以理解为“互动媒体 / 广告技术规则”。

它主要看：

- digital advertising
- adtech
- social media
- content platform
- streaming platform

例如：

- `Digital media and entertainment`

如果描述中还有广告、流量变现、内容平台、创作者生态等词，命中的稳定性就会更高。

### 4.5 当前并没有一个真正叫 `cloud_infrastructure` 的独立规则键

这一点很重要，答辩时最好说实话。

当前代码里并没有单独定义一个名为 `cloud_infrastructure` 的最终细分规则。更接近它的，是：

- 软件与 IT 服务这个大方向；
- 以及 `data_processing_and_outsourced_services` 这类更偏基础设施/托管方向的细分规则。

也就是说，像 `Cloud and AI infrastructure` 这种名字，当前系统会识别到“这可能和 IT 基础设施相关”，但是否能稳定落到很细的分类，还取决于描述里有没有更多证据。

## 5. 哪些业务线容易直接分类

从当前已经实现的规则来看，下面几类业务线相对容易稳定命中。

### 5.1 电商零售 / 平台交易

典型特征词比较集中，比如：

- ecommerce
- online retail
- marketplace
- merchant platform

例子：

- `China commerce retail business`

这类业务线的商业模式特征比较明显，因此更容易直接落到零售 / 平台交易相关分类。

### 5.2 软件服务 / 企业软件

如果文本里出现了比较明确的软件行业词：

- SaaS
- ERP
- CRM
- workflow software
- developer tools

那么它通常比较容易被分到应用软件或系统软件相关方向。

例子：

- `Enterprise software services`
- `Developer tools platform`

### 5.3 金融科技 / 支付处理

支付类业务的词也比较稳定，例如：

- payment processing
- payment gateway
- merchant acquiring
- digital wallet
- point of sale

例子：

- `Payment processing services`

这类业务线往往容易进入金融服务下的支付处理方向。

### 5.4 半导体 / 半导体设备材料

半导体相关业务的行业词通常很强，例如：

- semiconductor
- wafer
- foundry
- fabless
- lithography
- deposition equipment

例子：

- `Semiconductor manufacturing`
- `Semiconductor equipment`

这类业务线通常比“平台服务”“解决方案”之类的泛化名称更容易稳定分类。

### 5.5 科技硬件 / 通信设备

如果名称里出现：

- smartphone
- tablet
- server
- storage
- router
- switch

那么系统比较容易判断它属于科技硬件或通信设备方向。

### 5.6 能源设备、可再生能源、电力

当前代码已经覆盖了：

- 可再生发电
- 电力公用事业
- 电池与储能设备
- 一体化油气

例如：

- `Battery storage equipment`
- `Electric utility services`

但如果名称只写 `Energy solutions`，就很可能不够稳定。

## 6. 哪些业务线会保守处理

当前系统最容易“收着分”的，不是那些行业词很强的业务，而是那些名字听起来很大、很现代、但信息量很低的业务线。

### 6.1 只看起来高级，但不够具体的名称

例如：

- `Platform services`
- `Enterprise solutions`
- `Digital ecosystem`
- `Integrated services`

这些词的问题在于：几乎可以出现在很多行业里。

比如 `Platform services` 既可能是电商平台，也可能是支付平台、SaaS 平台、广告平台，甚至是物流平台。只看名字，很难稳妥地下结论。

### 6.2 “Cloud and AI infrastructure” 这类跨边界表达

这类业务线的难点在于，它同时包含：

- cloud
- ai
- infrastructure

看起来很技术，但并不能直接说明它到底是：

- 应用软件
- 系统软件
- 数据中心 / 托管服务
- 还是更宽泛的技术平台

所以当前实现里，这类业务线经常会：

- 只停在较粗层级；
- 或进入 `pending`；
- 或进入 `needs_llm_review`。

### 6.3 物流、履约、供应链类表述当前覆盖还不完整

例如：

- `Cainiao logistics services`

当前规则里，`fulfillment`、`delivery` 这类词更多是作为数字商业方向的辅助词存在，而不是一个独立、完整的物流规则家族。

这意味着：

- 如果业务线文本里同时出现零售、电商、商家平台等词，它可能被理解为电商生态的一部分；
- 但如果它更像纯物流 / 供应链服务，当前规则未必能稳定细分；
- 因此这类业务线在现阶段更容易落入保守输出或未映射状态。

### 6.4 工业设备类当前也不是重点覆盖对象

例如：

- `Industrial equipment`

这个名称很像一个真实行业，但当前这套规则主干更偏数字经济、科技、金融科技、能源、医疗科技等方向，并没有做一整套工业设备规则家族。

所以这类业务线如果只有很宽泛的名字，当前系统更可能给出：

- `unmapped`
- 或者一个较谨慎的待补判结果

而不会假装自己已经有很成熟的工业设备细分能力。

## 7. 当前几种状态分别是什么意思

当前前后端都会显示一些状态名。它们不是简单标签，而是系统对“当前结果是否稳”的判断。

### 7.1 `confirmed`

意思是：规则证据比较充分，系统认为可以正式确认当前分类结果。

通常出现的原因是：

- 业务线名称或别名里有明显行业词；
- 命中规则较集中；
- 没有强冲突候选；
- 分数达到了确认阈值。

例子：

- `China commerce retail business`
- `Semiconductor manufacturing`

### 7.2 `pending`

意思是：大方向基本对，但系统对最细层级还不够放心，所以先保守保留。

它常见于这种情况：

- 已经知道大概属于哪个方向；
- 但四级分类证据还不够强；
- 系统宁愿先停在三级，不直接下钻到底。

例子：

- `Cloud and AI infrastructure`

如果它能大致看出属于 IT 服务或基础设施方向，但叶子层级还不稳，就可能进入 `pending`。

### 7.3 `needs_llm_review`

意思是：当前规则还不够，需要更多语义理解或更丰富上下文，适合交给模型辅助判断。

进入这个状态的常见原因有：

- 名称过于泛，比如 `Platform services`
- 描述过于空，比如只有一个大词，没有行业特征
- 规则识别到是新兴业务、边界业务
- 虽然像某个方向，但细分证据太弱

例子：

- `Enterprise solutions`
- `Digital ecosystem`
- `Cloud and AI infrastructure`

### 7.4 `conflicted`

意思是：当前至少有两个候选方向都很像，而且分数差距太小，系统不敢强判。

这类情况常见于：

- 同时带有多个行业方向的词；
- 或者同一家族里两个细分规则竞争得很接近。

例子：

- 一条业务线既像支付平台，又像广告/流量变现平台；
- 或者一个名称同时命中两个非常接近的候选。

这时系统宁可承认“有冲突”，也不随便选一个。

### 7.5 `unmapped`

意思是：当前规则主干没有找到稳定匹配。

它不一定代表这条业务线没有行业属性，而更常见的意思是：

- 当前已实现规则还没覆盖到；
- 或者现有文本太少，根本不足以判断。

例子：

- `Industrial equipment`
- 某些纯物流、供应链、综合服务类业务线

### 7.6 `needs_manual_review`

这个状态在规则主流程里不是主产物，它更多出现在 LLM 辅助结果或人工流程里。

可以把它理解为：

- 系统已经形成一个候选建议；
- 但这一步不想直接自动生效；
- 需要人工再确认一次。

答辩时可以这样讲：当前规则刷新主要产出的是 `confirmed / pending / needs_llm_review / conflicted / unmapped`，而 `needs_manual_review` 更多是后续人工工作流中的状态。

## 8. `mapping_basis` 到底是什么

`mapping_basis` 可以理解成“系统为什么这么分”的简要留痕。

它不是给普通用户直接看的自然语言结论，而是给研究、审阅、调试、后续人工确认用的过程摘要。

### 8.1 自动 `mapping_basis` 会记录什么

当前实现里，它通常会记录几类内容：

- 最终决策是什么，例如 `confirmed`、`pending`
- 命中了哪些规则
- 哪些词在名称、别名、说明、公司上下文中被命中
- 是否出现了负向信号
- 当前只保留到了第几层
- 有没有额外备注，例如“为了安全先不下钻叶子层级”

一个典型形式像这样：

```text
decision=pending | rules=data_processing_and_outsourced_services | hits=name[cloud] description[data center, managed services] company[technology] peer[] | negatives=[] | depth=level_3 | comment=leaf withheld for safety
```

这串文本的本质不是“展示给前端用户的最终文案”，而是系统内部的可解释依据。

### 8.2 为什么系统要保留它

因为产业分类不是完全确定性的任务。很多时候，研究人员会问：

- 你为什么把它分到这里？
- 你依据的是名称、说明，还是公司背景？
- 你为什么没分到更细一级？

`mapping_basis` 就是为了回答这些问题。它相当于规则分类过程里的“留痕说明”。

### 8.3 它和人工修订依据有什么区别

二者不要混淆。

- 自动 `mapping_basis`：是系统自己生成的规则依据
- 人工修订依据：是研究人员手动填写的修改理由

前者回答“系统为什么这么判断”，后者回答“人工为什么要改系统结果”。

这两个字段的语义不同，所以在前端展示时也应该分开。

## 9. 一个完整例子

下面用两个典型业务线，演示当前系统是怎么走的。

### 9.1 例子一：`China commerce retail business`

#### 第一步：系统先读文本

它会先看：

- `segment_name`：China commerce retail business
- `segment_alias`：如果有，也一起看
- `description`：如果有，会补充判断
- 公司简介和同公司其它业务线：作为背景参考

#### 第二步：做标准化

像 commerce、retail、business 这些词会被统一成小写、去符号后的可比较形式。

#### 第三步：判断大方向

这里的 `commerce`、`retail` 很容易让系统把它归入“数字商业 / 零售平台”方向。

#### 第四步：看更细规则

如果还有：

- online retail
- marketplace
- ecommerce
- merchant platform

这类词，就更容易命中 `broadline_retail_marketplace` 这条细分规则。

#### 第五步：决定状态

如果证据集中、冲突不大、分数达到确认阈值，系统通常会直接给出：

- 零售平台相关的四级分类
- 状态为 `confirmed`

这类例子很适合答辩时说明“为什么有些业务线可以比较稳定地自动分类”。

### 9.2 例子二：`Cloud and AI infrastructure`

#### 第一步：系统先读文本

这个名字看上去很“技术”，但问题也正出在这里：它很强，却不够细。

#### 第二步：标准化

系统会把：

- cloud computing -> cloud
- artificial intelligence -> ai

这类写法做统一。

#### 第三步：判断大方向

它大概率会被系统看成和软件 / IT 服务 / 基础设施相关。

#### 第四步：为什么不一定直接细分

难点在于：

- `cloud` 可以指 SaaS，也可以指基础设施服务；
- `AI` 可能是产品能力，也可能只是宣传词；
- `infrastructure` 也可能覆盖数据中心、托管服务、平台底座等多种含义。

也就是说，系统能感觉到“方向像”，但不一定能稳稳地说“就是哪一个四级行业”。

#### 第五步：可能的结果

当前实现里，这类业务线常见的结果是：

- 停在较粗层级；
- 或进入 `pending`；
- 或进入 `needs_llm_review`。

这正体现了当前系统的设计原则：宁可先给一个保守结果，也不为了显得“智能”而乱分到最细层。

## 10. 当前规则的优点和局限

### 10.1 优点

第一，这套规则已经不是“字符串硬匹配”那么简单，而是做了多源文本综合判断。

它会一起看：

- 业务线名称
- 业务线别名
- 业务说明
- 公司上下文
- 同公司其它业务线

第二，它采用了“先家族、后细分”的两层结构，这比直接拿一个词硬套行业要稳得多。

第三，它强调保守输出。对于研究型系统来说，这一点很重要。因为产业分类看似像标签问题，实际上经常带有边界性和解释性。如果系统乱分，很容易误导后续分析。

第四，它保留了 `mapping_basis`，便于回看和解释。这对于毕业设计来说也是一个亮点，因为它体现了“可解释的规则系统”，而不是一个只给结果、不说理由的黑箱。

### 10.2 局限

当前规则虽然已经覆盖了电商、媒体、金融科技、软件、半导体、硬件、能源、医疗科技等一批主干方向，但覆盖面还远远不是“全行业完整”。

例如：

- 物流 / 供应链目前没有独立完整家族
- 工业设备不是当前重点覆盖方向
- 很多跨界平台型业务仍然只能保守处理
- 一些名称过于泛化时，规则很难仅凭几个词稳定落到最细层级

此外，当前很多判断还是基于短文本关键词和上下文线索，缺少更深层的商业模式理解。这也是为什么系统后续仍然需要：

- 人工征订 / 人工修订
- LLM 辅助补判

可以把当前版本理解为：它已经搭起了第一版“规则主干”，能把一批典型业务先稳稳分出来；但对于边界业务、新业务、模糊业务，仍然需要更强的语义判断和人工把关。

## 11. 一段适合论文 / 答辩引用的总结表述

本项目在产业分析模块中，构建了一套面向业务线文本的规则映射机制。系统以公司业务线名称、别名、业务说明及公司上下文为输入，先完成文本标准化，再通过“行业家族识别 + 细分规则匹配”的两层判断，将业务线映射为结构化的 GICS 分类结果。对于证据充分的业务线，系统可直接输出较完整的分类层级；对于描述模糊、候选冲突或规则覆盖不足的情况，系统采用保守输出策略，仅保留较粗层级或转入待模型补判、候选冲突、未映射等状态，从而兼顾分类效率、结果稳定性与可解释性。

## 12. 一句话总结

如果要用一句最适合答辩的话来概括当前实现，可以这样说：

这套系统不是“看到一个词就强行贴标签”，而是先判断业务线大概属于哪个产业方向，再决定能不能稳妥地下钻到更细分类；能确认的就确认，不能确认的就保守保留，并把判断依据留痕下来，供后续人工和模型继续处理。
