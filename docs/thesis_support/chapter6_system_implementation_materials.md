# 第六章系统实现与功能展示文字素材

【系统开发与运行环境】

本系统采用前后端分离的方式完成工程实现，整体定位为面向企业国别归属与业务结构研究的分析型系统。后端主要基于 FastAPI 框架构建，利用 SQLAlchemy 进行关系型数据访问与对象映射，默认运行环境采用 SQLite 数据库，以便在论文实验、样本验证和本地部署场景下快速复现。该设计降低了系统运行门槛，同时能够满足企业基础信息、股权关系、控制分析结果、产业分类结果以及人工征订记录等多类数据的持久化需求。

前端采用 Vue3、Vite、Element Plus 和 ECharts 实现。Vue3 负责组件化页面组织，Vite 用于开发调试和构建，Element Plus 提供表格、标签、表单、抽屉和提示等通用交互组件，ECharts 用于展示业务结构、控制关系和比例类图表。系统运行时，后端负责读取数据库、执行控制链分析和产业分类汇总，前端通过统一的数据请求获取分析结果并进行可视化展示。

从开发环境看，系统并未追求复杂的分布式部署，而是围绕本科毕业设计的研究验证目标，选择了较轻量、易维护、便于调试的技术组合。SQLite 适合作为样本数据库和研究型数据底座，FastAPI 便于组织接口与数据模型，Vue 前端则能够较直观地呈现分析过程和人工复核入口。整体上，系统更强调业务逻辑闭环、数据流转清晰和分析结果可追溯，而不是构建工业级平台或全自动决策系统。

【后端接口与数据服务实现】

后端数据服务围绕“企业综合分析”这一核心使用场景组织。系统将企业基础信息、控制链分析、国别归属判断、产业分析结果和人工征订记录分别存储在关系型表中，并通过后端服务进行统一聚合。前端在查询某一公司时，不需要分别理解各类底层表结构，而是通过综合分析结果获取页面展示所需的数据，包括公司基本信息、控制主体、控制路径、国别归属、业务线结构、产业标签和质量提示等。

在控制链分析方面，后端会基于股权关系和主体信息进行多层穿透，识别直接控制关系和实际控制主体，并对 joint、nominee 等复杂关系进行保守处理。当控制链无法形成明确结论时，系统会使用注册地回退等机制保证页面仍能提供可解释的结果。国别归属模块则在控制主体识别的基础上读取控制主体国家或地区信息；在缺少有效控制主体时，回退至企业注册地，从而形成较完整的国别归属输出。

产业分析部分采用事实层与结果层分离的方式。业务线事实保存于 business segments 数据中，产业分类结果保存于分类结果层。后端根据业务线文本和规则映射生成 GICS 分类建议，同时保留 review status、confidence、mapping basis 等字段，用于表达结果的不确定性和复核依据。人工征订、LLM 辅助建议和规则结果之间通过服务层进行协调，正式写回结果会保留操作痕迹。由此，后端不仅提供数据读取能力，也承担了自动分析、结果汇总、质量检查和人工确认之间的协作职责。

【公司综合分析页面实现】

公司综合分析页面是前端展示系统能力的主要入口。页面以公司为分析对象，围绕“控制结构、国别归属、业务结构”三个层面组织信息。用户可以通过公司名称、股票代码或指定编号查询企业，前端获取后端综合分析结果后，将公司基本信息、控制链结论、实际控制国别、业务线结构和产业分类结果集中展示，形成面向研究人员的一站式分析界面。

页面实现上，前端并未将各模块割裂为彼此独立的查询入口，而是通过综合数据流将多个分析模块连接起来。后端返回的 summary 类数据承担了页面数据底座的作用，前端在此基础上进行模块化渲染：顶部区域展示公司概况和核心判断结果，中部区域呈现控制关系和控制链路径，产业分析区域则展示业务线分类、收入或利润结构、待复核状态和质量提示。这样的组织方式便于研究人员从总体结论逐步下钻到证据和明细。

在交互设计上，页面强调分析过程的可阅读性，而不是仅给出一个最终标签。控制链和国别归属结果会配合关系明细展示，产业分析结果会显示业务线类型、产业层级和复核状态。对于规则分类存在不确定性的业务线，页面通过状态标签、提示信息和人工确认入口引导用户进一步核验。公司综合分析页面因此承担了“自动分析结果展示”和“人工征订工作台入口”两类功能，使系统能够服务于论文中的研究验证流程。

【控制链与控制关系展示实现】

控制链与控制关系展示模块主要用于说明企业实际控制主体的形成过程。后端根据股东结构、控制比例、控制关系类型和多层穿透规则，构建企业与股东实体之间的控制网络，并识别直接控制主体和实际控制主体。对于能够形成明确链路的样本，系统会给出控制路径、控制强度和控制类型；对于 nominee、joint 或信息不足的情况，系统会通过保守规则和回退机制进行处理，避免将不稳定结果包装成确定结论。

前端展示时，将后端生成的控制关系转换为适合页面呈现的结构。一方面通过表格或摘要区展示控制主体、控制国别、控制比例和判断依据；另一方面通过关系图或路径视图展示企业与股东实体之间的层级关系。这样的展示方式能够帮助研究人员理解系统为何给出某一控制结论，而不是只看到单一的国别标签。

该模块的实现重点在于自动分析结果与人工修订结果的并存。自动控制链分析提供初始判断，人工征订模块则允许研究人员在特殊样本中补充或修正实际控制主体、控制国别和控制路径。前端在展示时会区分自动结果与人工确认结果，并在必要时提示当前结论来源。通过这种方式，控制链模块不仅实现了股权穿透和控制主体识别，也支持对复杂企业结构进行可解释、可追踪的人工复核。

【产业分析与业务线展示实现】

产业分析模块围绕企业业务结构展开，实现上采用业务线事实层和产业分类结果层分离的设计。业务线事实层记录企业披露或整理得到的业务线名称、别名、描述、业务线类型、收入占比、利润占比和报告期等信息；分类结果层则记录针对每条业务线生成的产业分类、主分类标记、复核状态、分类来源、置信度和规则依据。这样的分层设计有助于区分“企业实际业务线事实”和“系统对业务线的产业解释”。

规则型分类主要用于生成初步产业映射。系统会结合业务线名称、别名、描述、公司文本和同公司其他业务线文本，与预设的行业家族、关键词和层级规则进行匹配，形成 GICS 层级分类建议。由于业务线文本常常存在概括性强、跨领域表达和新兴业务边界不清等问题，规则结果不会被视为最终结论。系统通过 confirmed、pending、needs LLM review、conflicted、unmapped 等状态表达分类结果的稳定程度，并将不确定样本暴露给后续辅助判断和人工确认。

前端产业分析区域会按 primary、secondary、emerging 和 other 展示业务线结构，同时呈现产业标签、收入或利润占比、复核状态和质量提示。质量检查会提示缺少主营业务线、业务线无分类、主营业务线无分类、重复业务线名称和同一业务线多个主分类等问题。Industry Workbench 在系统中定位为临时分析和辅助工具，可用于输入或调整业务线并快速查看规则或 LLM 建议，但它不是正式主流程，正式结果仍以数据库中的业务线与分类结果为准。

【人工征订与辅助确认交互实现】

人工征订与辅助确认机制是系统实现中保证结果可靠性的重要环节。由于企业控制结构和业务线分类都存在信息披露不完整、概念边界模糊和样本差异较大的情况，系统没有将自动分析结果直接作为最终结论，而是为研究人员提供了人工确认、人工覆盖和辅助建议的交互入口。自动分析负责提供初步结果，人工征订负责在复杂或不确定场景下补充研究判断。

在控制归属场景中，人工征订可以用于修正或确认实际控制主体、实际控制国别和控制路径。前端通过表单、选择器和路径构建交互收集人工输入，后端将人工结果写入专门的数据结构，并在综合分析页面中优先展示当前生效结果。对于仅有名称快照、需要新建主体或绑定已有主体的情况，系统也提供了不同录入方式，以适应研究数据整理中的多种情况。

在产业分类场景中，LLM 仅作为辅助建议机制使用。系统可以针对某条业务线生成分类建议，并将规则结果、公司上下文和同公司业务线作为参考信息提供给模型。模型输出不会自动成为正式结果，仍需要前端展示给研究人员确认；确认后才会写回分类结果层。人工分类则要求研究人员填写分类层级和依据，写回后标记为人工结果。相关操作会通过 annotation logs 记录前后变化、操作原因和操作者信息。由此，系统形成了“规则初筛、LLM 辅助、人工确认、日志留痕”的闭环，更适合作为研究型征订系统，而不是全自动分类平台。
